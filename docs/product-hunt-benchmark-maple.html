Story spine

The launch should be understood in ten seconds.

Product Hunt visitors will scan the thumbnail, tagline, gallery, and first comment quickly. Everything should repeat one idea from different angles: existing AI subscription, local workspace, reviewable wiki.

  1. Start with the cost insight. Students already pay for ChatGPT or Claude. Maple should not feel like yet another AI bill.
  2. Show the transformation. Messy PDFs, notes, links, and screenshots become summaries, concepts, guides, and an index.
  3. Make trust concrete. Sources stay separate. AI writes are shown as changed files. Users can review and undo.
  4. Anchor the user. This is for non-technical learners who do not want terminal workflows, API billing, or prompt engineering.

The launch thesis

Maple's novelty is not "AI makes notes." The launch has to make the bundle obvious:

  • existing ChatGPT/Codex/Claude subscription,
  • local folder ownership,
  • wiki-native durable artifacts,
  • review/undo safety for AI writes,
  • non-technical learner UX.

Launch-day ops

Do not improvise the first six hours.

Product Hunt is a conversation surface. The launch page gets attention, but replies convert confusion into clarity.

"Does Maple upload my documents?" Answer with the actual product behavior. Distinguish local storage from what the user's chosen AI provider may receive during a build.
"Do I need API keys?" Lead with the MVP goal: use the user's existing ChatGPT/Codex or Claude-style subscription path first, not API billing.
"Is this just NotebookLM?" Say: NotebookLM is a source-grounded research assistant; Maple is a local folder workspace that creates reviewable wiki artifacts.
"Where is Windows?" Be direct: macOS first for the MVP; Windows after the core source-to-wiki workflow proves useful.
"Can I trust AI-generated wiki pages?" Do not overpromise. Emphasize reviewable changed files, sources kept separate, and undo.
